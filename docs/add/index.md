# 数字资产云技术架构

## 当前 CLI 架构（V1.0 已实现）

当前 CLI 采用三层解耦设计，各层职责清晰，通过 dataclass 传递数据，可独立测试：

```
cli.py（入口层）
    │  解析命令行参数
    │  调用 planner → file_operator
    │  格式化输出（颜色/摘要）
    ▼
planner.py（配置层）
    │  加载 contracts.yaml
    │  扫描 journal 目录
    │  生成 Workflow { ArchiveTask }
    ▼
file_operator.py（操作层）
       逐文件移动（copy2 + unlink）
       失败自动回滚
       清理空源目录
```

各层设计原则：
- **入口层**不直接操作文件系统，只负责编排和展示
- **配置层**不执行写操作，纯数据转换，不依赖操作层
- **操作层**不依赖 YAML / CLI 参数，只接收具体 `Path`，返回结构化结果

详见 `src/cli/docs/` 下各模块文档。

## 架构原则

1. **核外挂载，内核稳定**：内核只做三件事——资产建模、版本快照、事件路由。所有"飞书转换器"、"GitHub转换器"做成独立插件
2. **模块化设计**：功能模块化，便于扩展和维护
3. **通用化抽象**：通过抽象概念和通用规则实现通用性和可复用性
4. **Schema演进管理**：平台定义一套中间标准Schema（ISDL - 统一数字资产描述语言），版本控制的是ISDL而非底层存储

## V2.0 规划：引擎化演进

### 数据处理流程

1. 中间标准Schema（ISDL）：飞书数据转成ISDL，GitHub数据转成ISDL。版本控制控的是ISDL的版本，而非底层数据库DDL
2. 内容寻址版本控制：借鉴Git底层设计，通过哈希值标识资产版本，支持diff、回滚、fork
3. 事件溯源：记录资产每次变更的事件，通过回放事件得到最新状态
4. 最终一致性：跨平台不做"强一致同步"，而是基于事件驱动的最终一致，通过异步消息队列处理跨平台状态同步

### 演进路径

**Step 1: 契约驱动（✅ 已完成）**
- transform 定义从 YAML 而非硬编码
- `contracts.yaml` 读取契约配置

**Step 2: 抽取通用引擎（规划中）**
```
读取契约 → 遍历文件 → 执行 transform → 写入 → 记录日志
```
抽取为 `engine.py`，支持可插拔的 transform 链

**Step 3: 适配器抽象（规划中）**
- `LocalFSAdapter`（✅ 已验证）
- `FeishuAdapter`
- `GithubAdapter`

**Step 4: 事件溯源（规划中）**
- 每个执行步骤记录结构化事件
- 可通过回放事件重建状态
