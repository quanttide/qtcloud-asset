# 架构决策记录

> 记录当前 CLI 实现与原始 BRD / PRD / ADD 设计蓝图之间的差异及决策结论。
> 最后更新：2026-04-12

## 决策分类说明

| 标记 | 含义 |
|:---|:---|
| ✅ 已接受 | 经评估，该偏离为有意简化，当前无需对齐文档 |
| ⏸️ 推迟 | 未来可能需要，但当前版本不实现 |
| ❌ 不需要 | 确认永远不会实现，文档相关章节作废 |
| 🔧 已修复 | 测试或代码已调整完毕 |

---

## Q001: CLI 只有一个 `archive` 命令，覆盖面过窄

| 维度 | 说明 |
|:---|:---|
| **现状** | CLI 仅实现 `archive` 一个命令（journal → archive 文件移动） |
| **蓝图期望** | BRD 描述了多个业务场景——跨平台迁移、AI 提炼路线图、成熟度判断、审批流程等 |
| **决策** | ✅ **已接受**。当前版本定位为 **个人知识库归档工具 MVP**，聚焦核心文件操作闭环。跨平台、AI 提炼等能力作为远期路线图，不在 V1 实现。 |
| **来源** | `docs/prd/workflow.md`、`docs/brd/index.md` |

---

## Q002: 契约是"哑"的路径映射，缺少状态机和 ISDL

| 维度 | 说明 |
|:---|:---|
| **现状** | `contracts.yaml` 仅包含 `name`、`version`、`paths.journal`、`paths.archive` 四个字段 |
| **蓝图期望** | PRD 定义契约包含状态机、成熟度评分、治理语言记录、CID 内容寻址、`forked_from` 血缘 |
| **决策** | ⏸️ **推迟**。当前仅需支持单用户手动归档，状态机与评分体系缺乏实际触发场景。**V2 若引入自动化触发器时，再按 ISDL Lite 规范扩展契约格式。** |
| **来源** | `docs/prd/contract.md`、`docs/add/workflow.md` |

---

## Q003: 没有 `engine.py`，架构偏离 ADD 演进路径

| 维度 | 说明 |
|:---|:---|
| **现状** | `planner.py` 和 `file_operator.py` 直接耦合，没有独立的执行引擎 |
| **蓝图期望** | ADD 定义演进路径 Step 2 为"抽取通用引擎 `engine.py`" |
| **决策** | ✅ **已接受**。经代码审查确认：在单一命令、单适配器（本地文件系统）的场景下，`planner` + `file_operator` 的职责划分已足够清晰。`engine.py` 为当时预设的过度抽象，**决定不予实现**。若未来引入多适配器或多阶段流水线，再考虑抽取编排层。 |
| **来源** | `docs/add/index.md` |

---

## Q004: 没有适配器抽象

| 维度 | 说明 |
|:---|:---|
| **现状** | 所有操作硬编码本地文件系统 |
| **蓝图期望** | ADD 规划 `LocalFSAdapter`、`FeishuAdapter`、`GithubAdapter` 等插件化适配器 |
| **决策** | ⏸️ **推迟**。飞书 / GitHub 跨平台迁移为 BRD 中的高价值场景，但 **V1 先验证本地归档核心流程**。`file_operator.py` 已保留接口扩展点（`_copy`, `_delete` 方法），未来可无痛重构为适配器模式。 |
| **来源** | `docs/add/index.md`、`docs/prd/workflow.md` |

## Q005: 缺少 `roadmap` 命令（AI 提炼产品路线图场景）

## Q006: `src/provider` 为空壳，测试 import 路径不一致

| 维度 | 说明 |
|:---|:---|
| **现状** | `src/provider/app/` 下仅存空文件；测试曾引用错误路径 |
| **蓝图期望** | 原计划支持服务端模式 |
| **决策** | ✅ **已接受**。经评估，CLI 单机模式完全满足个人使用需求，服务端开发计划搁置。**保留 `src/provider` 空目录作为命名空间占位符**，避免未来扩展时忘记预留包结构。测试路径已于 `tests/` 中修正为 `src.cli.app` 引用。 |
| **来源** | `src/cli/tests/` |

## Q007: 缺少触发器机制

| 维度 | 说明 |
|:---|:---|
| **现状** | 只能手动执行 `cli archive` |
| **蓝图期望** | ISDL Lite 规划了 manual、cron、event 三种触发方式 |
| **决策** | ⏸️ **推迟**。手动触发已覆盖当前核心场景（"我想归档了，就运行一下"）。**V2 若需要无人值守自动归档，可结合系统 cron 或 launchd 实现，无需修改核心代码。** |
| **来源** | `docs/prd/workflow.md` |

---

## Q008: 缺少事件溯源 / 执行日志

| 维度 | 说明 |
|:---|:---|
| **现状** | `ArchiveResult` 仅返回内存统计，不持久化 |
| **蓝图期望** | 结构化执行日志，包含 `correlation_id` 等字段 |
| **决策** | ⏸️ **推迟**。当前个人使用场景下，终端输出已满足"知道干了什么"的需求。**V2 若涉及多人协作或合规审计，可引入 `structlog` 写入 `.jsonl` 文件。** |
| **来源** | `docs/add/index.md` |

## 总结：V1 实现范围边界

| 模块 | V1 状态 | 备注 |
|:---|:---|:---|
| 本地文件归档 (`archive` 命令) | ✅ 已实现 | 支持 `—delete` 标志控制源文件删除 |
| 契约驱动 (`contracts.yaml`) | ✅ 最小实现 | 仅 `paths.journal` / `paths.archive` |
| 跨平台适配器 | ⏸️ 推迟 | 硬编码本地文件操作 |
| 触发器 / 自动化 | ⏸️ 推迟 | 手动触发 |
| AI 路线图提炼 | ⏸️ 推迟 | 脚本验证，未集成 |
| 执行日志 / 溯源 | ⏸️ 推迟 | 内存统计，无持久化 |
| 服务端模式 | ❌ 不需要 | 空壳保留占位 |

**V1 核心价值主张：**
提供一个**简单、可靠、契约驱动**的本地文件归档命令行工具，解决个人知识库从"草稿区"到"归档区"的手动整理痛点。
