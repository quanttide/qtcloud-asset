# 数字资产云蓝图

## 产品定位

- 核心定位：打破数字资产的平台孤岛，用工程化手段（版本控制、显性化契约）管理个人和组织的知识资产
- 关键价值点：隐性工作流显性化，引入版本控制降维打击现有知识库工具

## 核心价值

- 隐性工作流显性化：将"我在飞书写文档 -> 提炼为GitHub Issue -> 同步到Jira"的隐式过程，变成可视化的、可执行的数字资产契约流水线
- 版本控制降维打击：现有知识库只有"历史记录"，没有真正的版本控制（无法diff、无法回滚、无法fork）。将Git理念引入非代码类数字资产

## 核心设计

### 原子定义

1. 数字资产"第二大脑"：定义一种抽象概念来表示数字资产的管理逻辑
2. 通用规则：制定通用规范和流程，使数字资产治理范式可复用

### 数字资产边界（V1.0）

V1.0对"数字资产"做严格狭义定义，只处理结构化/半结构化数据：

- 多维表格
- JSON Schema
- 标准 Markdown
- 任务清单

不碰非结构化的富文本（如带复杂排版的文档），避免跨平台转换复杂度指数级增长。

### 功能设计

- 跨平台转换功能：实现不同平台之间的数字资产互转，如飞书到 GitHub
- 个人外脑支持：管理多个数字资产的生命周期，简化工作流
- 标准数字资产定义：定义和维护各类数字资产的标准和规范

## 数据处理流程

1. 中间标准Schema（ISDL）：飞书数据转成ISDL，GitHub数据转成ISDL。版本控制控的是ISDL的版本，而非底层数据库DDL
2. 内容寻址版本控制：借鉴Git底层设计，通过哈希值标识资产版本，支持diff、回滚、fork
3. 事件溯源：记录资产每次变更的事件，通过回放事件得到最新状态
4. 最终一致性：跨平台不做"强一致同步"，而是基于事件驱动的最终一致，通过异步消息队列处理跨平台状态同步

## 架构原则

1. 核外挂载，内核稳定：内核只做三件事——资产建模、版本快照、事件路由。所有"飞书转换器"、"GitHub转换器"做成独立插件
2. 模块化设计：功能模块化，便于扩展和维护
3. 通用化抽象：通过抽象概念和通用规则实现通用性和可复用性
4. Schema演进管理：平台定义一套中间标准Schema（ISDL - 统一数字资产描述语言），版本控制的是ISDL而非底层存储

## MVP 演进路线

### Phase 1: 本地优先的"资产Git"

- 形态：本地客户端或 VS Code 插件
- 功能：本地管理数字资产（Markdown、JSON、CSV），提供可视化界面的 Git 操作（diff、branch、merge、rollback）
- 价值验证：验证"第二大脑+版本控制"是否真的有用户买单

### Phase 2: 单向的"资产管道"

- 形态：Phase 1 基础上加入"连接器"
- 功能：配置 Pipeline，把本地整理好的 JSON 任务清单一键推送到 GitHub Issues，或把本地 Markdown 发布到飞书文档
- 价值验证：验证跨平台转换需求真实性，以及格式映射的损耗用户能否接受

### Phase 3: 双向同步的"云蓝图"

- 形态：SaaS 云平台
- 功能：双向同步、Webhook 监听平台变更、团队协作、云端版本库
- 价值验证：蓝图描绘的终极形态

## 验证方向

1. 具体场景验证：把飞书多维表格里的 Bug 追踪表同步为 GitHub Issues，并在本地保留两个平台的修改历史
2. Schema定义验证：验证ISDL能否覆盖目标场景的数据结构
3. 格式映射验证：验证跨平台转换的格式损耗用户能否接受

## 困惑与挑战

1. 数据一致性 -> 通过最终一致性与状态机解决，为每个资产定义状态机（草稿 -> 已发布 -> 已归档），记录同步日志
2. 用户需求变化 -> 通过插件化架构解决，需求变化只需新增插件，内核不动
3. 格式丢失处理 -> V1.0严格限定资产边界，不碰复杂富文本，降低转换复杂度
