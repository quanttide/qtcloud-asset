# 数字资产云蓝图

## 产品定位

- 核心定位：打破数字资产的平台孤岛，用工程化手段（版本控制、显性化契约）管理个人和组织的知识资产
- 关键价值点：隐性工作流显性化，引入版本控制降维打击现有知识库工具

## 核心价值

- 隐性工作流显性化：将隐式工作过程变成可视化的、可执行的数字资产契约流水线（详见 contract.md）
- 版本控制降维打击：现有知识库只有"历史记录"，没有真正的版本控制（无法diff、无法回滚、无法fork）。将Git理念引入非代码类数字资产

## 核心设计

### 原子定义

1. 数字资产"第二大脑"：定义一种抽象概念来表示数字资产的管理逻辑
2. 通用规则：制定通用规范和流程，使数字资产治理范式可复用

### 数字资产边界（V1.0）

V1.0对"数字资产"做严格狭义定义，只处理结构化/半结构化数据：

- 多维表格
- JSON Schema
- 标准 Markdown
- 任务清单

不碰非结构化的富文本（如带复杂排版的文档），避免跨平台转换复杂度指数级增长。

### 功能设计

跨平台转换功能实现不同平台之间的数字资产互转，如飞书到 GitHub。个人外脑支持管理多个数字资产的生命周期，简化工作流。标准数字资产定义维护各类数字资产的标准和规范。

成熟度评分帮助判断资产是否达到可共享状态。评分来源包括手动打分和系统自动统计，与工作流触发器结合实现自动迁移提示。

治理语言解析记录自然语言描述与 ISDL 配置的对应关系。V1.0 先支持简单模式匹配，人工确认后执行。

## 已验证

- 本地脚本管理 Markdown 资产：通过 `backup_product_journal.py` 验证了资产的采集、移动、归档流程
- 本地 AI 转换管道：通过 `generate_product_roadmap.py` 验证了调用本地 Ollama 模型将日志转换为结构化蓝图的可行性
- 目录结构即资产模型：journal/archive/roadmap 三目录结构验证了"资产建模、版本快照、事件路由"的内核设计

## 困惑与挑战

1. 数据一致性 -> 通过最终一致性与状态机解决，为每个资产定义状态机（草稿 -> 已发布 -> 已归档），记录同步日志
2. 用户需求变化 -> 通过插件化架构解决，需求变化只需新增插件，内核不动
3. 格式丢失处理 -> V1.0严格限定资产边界，不碰复杂富文本，降低转换复杂度
