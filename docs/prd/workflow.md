# 数字资产工作流

## 故事一：契约执行引擎

### 用户故事
作为执行方，我想要系统自动执行数字资产契约定义的工作流，以便将静态契约变为可自动运行的跨平台流水线。

### 验收标准

**场景**：事件触发工作流
- **假设**：工作流配置了事件触发器
- **当**：指定平台资源发生匹配事件
- **那么**：系统启动工作流实例并执行定义的转换规则

**场景**：查看工作流状态
- **假设**：工作流实例正在运行或已完成
- **当**：执行方打开工作流监控页面
- **那么**：系统显示最近执行时间、执行结果、输入输出摘要

**场景**：处理执行失败
- **假设**：工作流执行过程中遇到错误
- **当**：系统检测到失败
- **那么**：工作流自动暂停，记录失败原因和上下文，通知相关负责人

### 业务规则

- 触发器支持事件触发、定时触发、手动触发、规则触发
- 规则触发当资产成熟度评分达到阈值时自动启动
- 工作流状态包括草稿、待审查、活跃、暂停、归档
- 执行失败时自动暂停，不会静默跳过
- 提供恢复或降级选项供用户处理失败实例

### 依赖

- 依赖适配器生态，需连接目标平台
- 依赖资产成熟度评分，支持规则触发

## 故事二：跨平台数据转换

### 用户故事
作为配置方，我想要定义数据从一个平台到另一个平台的转换规则，以便实现跨平台自动化流转。

### 验收标准

**场景**：配置转换规则
- **假设**：配置方需要打通飞书和 GitHub 的数据流转
- **当**：配置方选择源资产和目标资产并配置字段映射
- **那么**：系统保存转换规则并验证平台能力支持

**场景**：验证平台能力
- **假设**：配置方配置的工作流需要特定平台能力
- **当**：配置方保存工作流配置
- **那么**：系统检查所选平台是否支持所需能力并提示结果

**场景**：查看执行历史
- **假设**：工作流已执行多次
- **当**：配置方打开执行历史
- **那么**：系统显示每一步的输入、输出、执行时间和操作人

### 业务规则

- 每个适配器声明支持的资源类型、操作、事件
- 转换规则包含源资产、目标资产、参数映射、执行条件
- 配置时系统自动检查平台能力，避免配置无效工作流
- 系统定期检查适配器可用性，失败时暂停相关工作流并告警

### 依赖

- 依赖适配器注册，需有可用的平台连接器
- 依赖健康检查，确保平台连接稳定
