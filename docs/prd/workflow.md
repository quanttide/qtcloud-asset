# 数字资产工作流

## 产品定位

工作流是数字资产契约的执行引擎，让用户定义的契约从静态文档变为可自动运行的跨平台流水线。

## 核心价值

- 跨平台自动化：一次定义，自动在飞书、GitHub 等平台间流转数据
- 可视化编排：用户通过结构化配置定义工作流，无需编写代码
- 可审计执行：每一步执行都有记录，支持追溯和回放

## 工作流定义

用户通过 ISDL（统一数字资产描述语言）定义工作流，包含四个部分：

### 资产声明

用户声明工作流涉及的平台资源，例如：

- 飞书文档：声明需要的字段（文档ID、标题、URL）和约束条件（必须包含特定标签）
- GitHub Issue：声明需要的字段（仓库、标题、内容）

### 转换规则

用户定义数据如何从一个平台转换到另一个平台：

- 选择源资产和目标资产
- 配置转换动作（如"创建 Issue"）
- 设置参数映射（如飞书标题 -> Issue 标题前缀 + 原文标题）
- 定义执行条件（如仅当标签包含"紧急"时执行）

### 触发器

用户定义工作流何时启动：

- 事件触发：如飞书文档被打上特定标签时
- 定时触发：如每天固定时间执行
- 手动触发：用户手动启动

### 生命周期管理

用户管理工作流的运行状态：

- 草稿：编辑中，不执行
- 待审查：提交后等待审批
- 活跃：正常运行中
- 暂停：因错误或人工干预被阻断
- 归档：不再使用，保留历史记录

## 适配器生态

### 平台支持

工作流通过适配器连接不同平台，每个适配器声明自己支持的能力：

- 支持的资源类型（如飞书文档、GitHub Issue）
- 支持的操作（如创建、更新、打标签）
- 支持的事件（如文档创建、标签变更）

### 能力协商

用户在配置工作流时，系统自动检查所选平台是否支持所需能力，避免配置无效的工作流。

## 执行监控

### 运行状态

用户可查看每个工作流实例的当前状态：

- 最近一次执行时间
- 执行结果（成功/失败/暂停）
- 输入输出的摘要信息

### 失败处理

工作流执行失败时：

- 自动暂停，不会静默跳过
- 记录失败原因和上下文
- 通知相关负责人
- 提供恢复或降级选项

### 执行历史

用户可查看工作流的完整执行历史，包括每一步的输入、输出、执行时间和操作人。

## 健康检查

系统定期检查工作流依赖的平台适配器是否可用：

- 平台 API 是否可达
- 关键能力是否仍然存在
- 失败时自动暂停相关工作流并告警

## 使用场景

### 场景一：飞书文档同步 GitHub Issue

当飞书文档被打上"同步"标签时，自动创建 GitHub Issue，包含文档摘要和链接。

### 场景二：定时数据汇总

每天定时从飞书多维表格拉取数据，生成报告并发布到指定频道。

### 场景三：审批流

用户提交文档后，自动触发审批流程，审批通过后同步到目标平台。
