# AI 交付约束

## 故事一：交付自动验证

### 用户故事
作为验收方，我想要系统自动验证智能体交付的数字资产，以便无需逐字检查就能判断交付物是否达标。

### 验收标准

**场景**：自动验证交付物
- **假设**：智能体提交了一份 PRD 文档
- **当**：系统检测到新资产提交
- **那么**：系统加载对应资产类型的规则并执行验证

**场景**：查看验证结果
- **假设**：验证引擎完成检查
- **当**：验收方打开验证报告
- **那么**：系统显示总分、通过规则数、不通过规则数、警告数

**场景**：定位不通过项
- **假设**：验证报告中有不通过的项目
- **当**：验收方点击不通过项
- **那么**：系统高亮显示失败原因和建议修复方式

### 业务规则

- 验证结果分为通过、不通过、跳过三种状态
- 错误级别不通过时资产标记为不合格
- 警告级别不通过时标记为建议修改
- 验证结果包含规则名称、检查结果、失败原因、建议修复

### 依赖

- 依赖规则定义能力，需有可用的验收规则
- 依赖资产类型识别，需知道加载哪类规则

## 故事二：约束规则管理

### 用户故事
作为规则制定方，我想要定义和管理验收规则，以便适应不同资产类型和团队的验收标准。

### 验收标准

**场景**：创建自定义规则
- **假设**：规则制定方需要为新资产类型定义验收标准
- **当**：规则制定方基于模板创建规则并配置检查项
- **那么**：系统保存该规则并关联到对应资产类型

**场景**：查看规则版本历史
- **假设**：某规则经过多次修改
- **当**：规则制定方打开该规则的版本历史
- **那么**：系统显示每次变更的版本号、内容、日期、变更人

**场景**：回滚规则版本
- **假设**：当前规则版本存在问题
- **当**：规则制定方选择历史版本并执行回滚
- **那么**：系统恢复该版本的规则配置

### 业务规则

- 规则包含名称、描述、适用资产类型、验证条件、严重级别
- 验证条件分为格式检查、内容检查、关联检查三类
- 规则支持版本控制，每次变更都有记录
- 规则可 fork 创建变体，适应不同团队需求
- 资产验证时记录使用的规则版本，确保结果可追溯

### 依赖

- 依赖规则模板库，提供预定义的常用规则
- 依赖资产类型系统，规则需关联到具体资产类型
