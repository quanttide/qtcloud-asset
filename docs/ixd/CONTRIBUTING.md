# IXD 维护指南

## 文档定位

交互设计文档描述产品的界面布局和交互流程，不是视觉设计稿。

## 写作原则

### 描述界面布局

用文字描述清楚界面结构和元素位置。左侧是什么、右侧是什么、点击后发生什么。不用画图，用结构化文字表达。

### 聚焦交互而非视觉

不写颜色色值、字体大小、像素间距。写信息层级、操作反馈、导航结构。

### 用用户视角

从用户操作流程出发，描述用户看到什么、做什么、系统反馈什么。不是写组件属性和状态管理。

## 文档结构

采用以下结构：设计原则说明交互设计的核心指导思想，核心界面描述主要界面的布局和元素，交互流程描述用户完成关键任务的操作路径。

### 设计原则

写 3-5 条指导设计的原则，如极简主义、一致性、反馈即时。每条原则一句话描述，可附简短说明。

### 核心界面

每个核心界面单独起小节。描述界面布局（左右结构、上下结构、占比）、主要元素（按钮、表格、列表、卡片）、交互行为（点击、滑动、缩放）。

### 交互流程

从用户视角描述完成关键任务的完整路径。用箭头连接各步骤：用户做什么 -> 系统反馈什么 -> 用户下一步做什么。

## 与 PRD 的关系

PRD 定义做什么和为什么，IXD 定义用户怎么用。两者配合使用，PRD 是功能需求，IXD 是用户体验设计。

## 审查清单

提交 IXD 前自检：是否描述了所有核心界面、是否清楚表达了交互流程、是否避免了纯视觉描述、是否能让开发人员理解实现需求、是否与 PRD 中的功能描述一致。
