# 维护指南

## 文档结构

QA 目录按领域划分文件，每个文件对应一个独立的质量维度：

- `README.md` — 总览，列出所有 QA 文件及其状态
- `contract.md` — 契约体系的质量保证
- `workflow.md` — 治理工作流的质量保证

## 编写格式

每个 QA 文件遵循「准则 — 判定 — 证据」结构：

```markdown
# 领域名称质量保证

将每一项 QA 拆解为「准则 — 判定 — 证据」。

## 维度名称

准则：一句话总结底线。

### 判定项名称

- 现实需求：用业务语言描述要验证什么
- 判定标准：具体的通过条件
- 存证：测试文件、脚本输出或实际运行结果
- 状态：✅ 符合 / ⏸️ 待实现
```

## 格式规范

- 准则用一句话概括，放在维度标题下方
- 判定项用无序列表，不用表格
- 存证指向具体的测试文件、代码路径或运行结果
- 状态只使用两种：✅ 符合 或 ⏸️ 待实现

## 新增 QA 文件

当出现新的质量维度时（如部署安全、数据质量等），创建新文件：

1. 用描述性名称命名，如 `deploy.md`、`data-quality.md`
2. 在 `README.md` 目录中添加链接和状态
3. 按「准则 — 判定 — 证据」格式编写

## 更新现有 QA

当代码或测试发生变更时：

1. 检查受影响的判定项的存证是否仍然有效
2. 更新状态（符合或待实现）
3. 如果判定标准发生变化，同步更新描述
4. 如果整个维度不再适用，删除该维度

## 状态约定

- ✅ 符合：判定标准已满足，存证有效
- ⏸️ 待实现：判定标准已定义，但尚未实现或存证缺失
