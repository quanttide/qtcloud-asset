# 质量保证文档

## 质量目标

验证系统是否满足创始人的业务需求。创始人希望用自然语言描述治理意图，系统自动理解并执行迁移操作。创始人希望系统能判断资料成熟度，提示何时该迁移。创始人希望整个治理过程可追溯可审计。测试策略围绕这三个核心业务价值展开。

## 测试策略

测试金字塔分层保障业务价值。单元测试确保核心逻辑正确，契约解析正确理解治理意图，状态机正确执行审批流程。集成测试验证组件协作，适配器正确执行跨平台迁移，成熟度评分正确触发工作流。端到端测试验证完整业务流程，从创始人输入治理语言到系统完成迁移操作。

测试优先级按业务价值排序。契约解析优先验证系统是否能理解创始人的治理意图，这是系统入口必须可靠。执行器优先验证迁移操作是否按审批流程正确执行，这是核心业务流程必须完整无误。成熟度评分验证系统是否能判断资料是否达到可共享状态，这是创始人的核心痛点。适配器验证跨平台迁移是否能正确执行，这是业务场景的技术实现。

## 测试场景

**治理语言理解**

验证创始人用自然语言描述迁移意图时，系统是否能正确解析为契约配置。测试场景包括创始人说把这个方案迁移给团队，系统应生成迁移契约。创始人说这个方案还不成熟，系统应暂停迁移流程。创始人说回滚上次迁移，系统应能恢复到迁移前状态。

**成熟度判断**

验证系统是否能判断资料成熟度并触发迁移提示。测试场景包括资料成熟度评分达到阈值时，系统应提示创始人或自动触发迁移。资料成熟度评分低于阈值时，系统应不触发迁移。创始人手动调整成熟度评分时，系统应更新触发判断。

**迁移执行**

验证系统是否能正确执行跨平台迁移操作。测试场景包括从个人知识库迁移到团队知识库时，内容应完整迁移。迁移过程中权限应正确调整。迁移后团队应及时收到通知。迁移失败时应能回滚到原状态。

**审批流程**

验证迁移操作是否按审批流程执行。测试场景包括契约分叉后应进入待审查状态，必须人工审批。审批通过后契约应变为活跃状态开始执行。执行失败后契约应进入暂停状态，必须人工恢复。

**可追溯性**

验证整个治理过程是否可追溯可审计。测试场景包括每个资料的迁移历史应清晰可查。迁移决策依据应完整记录。执行日志应包含时间、操作人、输入输出。创始人应能回溯自己的治理决策。

## 测试结构

tests 目录下按 unit、integration、e2e、fixtures 组织。unit 单元测试验证契约解析、状态机、成熟度评分等核心逻辑。integration 集成测试验证适配器协作、工作流执行。e2e 端到端测试验证完整业务流程。fixtures 测试数据模拟创始人的知识库场景。

## 质量标准

代码覆盖率大于 70% 保障可维护性和变更信心。契约解析测试必须通过，保障系统入口可靠理解治理意图。状态机测试覆盖所有状态转换，保障审批流程完整无误。成熟度评分测试覆盖阈值触发场景，保障创始人核心痛点得到解决。适配器测试每个适配器至少 3 个用例，保障跨平台迁移能正确执行。

## CI/CD 集成

每次合并请求时运行测试，强制执行质量门禁，确保不符合测试标准的代码无法进入主分支。测试失败意味着业务需求未被满足，必须修复后才能合并。

## 测试数据管理

fixtures 使用最小化数据聚焦业务场景，不包含真实敏感信息保障安全性，目录结构映射创始人的知识库组织方式，如个人知识库 quanttide-founder 和团队知识库 quanttide-tech 的模拟场景。

## 故障排查

测试失败时分析对应的业务问题。契约解析失败意味着系统无法理解创始人的治理意图。状态机流转失败意味着审批流程不符合业务规则。适配器执行失败意味着跨平台迁移无法正确执行。成熟度评分失败意味着系统无法判断资料是否成熟。根据业务问题定位技术原因并修复。
