# BRD 写作指南

## 核心原则

BRD 只描述问题，不描述功能。把「要做什么」改造成「发生了什么」。

## 标准结构

每个场景必须包含三部分：

### 工作角色

定义场景中参与的人，用行为角色而非岗位名称。

正确：发起方、审核方、执行方、确认方
错误：产品经理、财务、CEO

### 问题

必须回答四要素：

1. 触发：什么情况下发生
2. 现状：现在是怎么做的
3. 困难：卡在哪里
4. 后果：导致了什么

### 假设

使用固定句式：如果存在这样一个平台，能够……

这是 BRD 到 PRD 的唯一接口，定义需求边界。

## 审查清单

- 非技术人员能看懂
- 包含真实人物和场景
- 说明业务动机和期待
- 具体到可以想象操作画面
- 清楚表达为什么重要
- 避免技术术语
- 不同人对问题理解一致
