# BRD 维护指南

## 文档定位

商业需求文档（BRD）是业务方与技术团队的沟通桥梁，不是技术规格文档。

## 写作原则

本文档基于[量潮科技写作格式标准](https://github.com/quanttide/quanttide-specification-of-business-entity/blob/v0.0.3/docs/format.md)。

### 讲业务故事

从真实业务场景出发，有背景、人物、痛点。创始人整理个人知识库时，发现某些技术方案已经过实践验证，团队成员经常来询问这些资料。他想共享给团队，但又担心过早公开导致版本混乱。而不是"系统需要支持仓库间迁移功能"。

### 用业务语言

避免技术术语，使用业务方能理解的表达。比如用"让不同团队独立开发和部署"代替"实现微服务架构"，用"让系统自动执行治理决策"代替"配置规则引擎"。

### 有情感和人性化

体现真实的人。业务方的期待是"希望能让团队用上成熟的方案"，担忧是"怕迁移后权限管理混乱"，动机是"从个人知识到组织资产的转化"。

### 具体化场景

用真实例子说明。包括真实的业务场景、具体的痛点、实际的操作流程、量化的业务目标。

### 回答"为什么"

每个需求都要回答为什么这个业务场景重要、为什么现在需要解决、为什么这个问题值得投入、解决后的业务价值是什么。

## 文档结构

建议采用以下结构：背景故事说明谁在什么场景下遇到了什么问题，业务场景描述具体的业务流程和操作，痛点分析当前存在的困难和代价，需求描述期望的解决方案，业务价值说明解决后的收益和影响，成功标准明确如何衡量是否解决。

## 审查清单

提交 BRD 前自检能否让非技术人员看懂、是否包含真实的人物和场景、是否说明了业务动机和期待、是否具体到可以想象操作画面、是否清楚表达了"为什么"、是否避免了纯技术术语。

## 常见错误

常见的错误包括写成技术规格文档、只有功能描述没有业务背景、使用抽象概念代替具体场景、缺少业务方的情感和动机、没有说明业务价值和优先级。

## 反馈与迭代

BRD 应该是活文档，需要与业务方保持沟通，根据反馈持续完善，记录需求演进过程，保持与实际业务的一致性。
