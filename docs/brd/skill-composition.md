# 原子技能编排

## 背景

当前的治理能力是以"场景"为单位硬编码的，一个迁移或归档场景就对应一整套固定逻辑。随着治理场景不断增多，系统缺少一层抽象，无法将可复用的基础能力从具体场景中剥离，导致难以灵活应对多变的业务需求。

创始人已经定义了若干原始技能（原子技能），这些技能是基础功能模块。创始人希望用户可以根据业务需求，灵活自定义工作流，而不是把所有治理逻辑都硬编码进系统。

## 业务场景

创始人希望把归档工作流从代码中解放出来。归档的本质只是"列举文件清单"、"移动文件"、"验收结果"这三个基础动作的有序组合。后面三个都是用代码写好的基础功能，归档则是需要用户额外定义的。

用户可以根据想选择的技能添加进工作流，并且可以自定义顺序。比如：

- 归档 = 列举文件清单 → 移动文件 → 验收结果
- 迁移 = 列举文件清单 → 复制文件 → 权限变更 → 验收结果
- 备份 = 列举文件清单 → 复制文件 → 压缩文件 → 验收结果

创始人希望系统不要把归档死写进代码里，而是提供这些现成的"积木"（原子技能），让他通过配置文件，按需挑选积木并排列顺序，像套公式一样自由拼装出治理流程。

## 痛点分析

新增或调整治理场景必须依赖开发。每增加一种工作流，都要从零编写代码，即使底层能力已在其他场景中实现过。用户无法根据业务变化快速试错，被固定的代码逻辑锁死了灵活性。

工作流定义与代码强耦合。当需要调整工作流顺序或添加新步骤时，必须修改代码、重新测试、重新部署，周期长、成本高。

原子技能无法复用。相同的"列举文件清单"、"移动文件"等能力，在不同工作流中重复实现，没有统一的能力抽象层。

## 需求描述

系统底层提供一组预定义的原子技能（如列举文件、移动文件、压缩文件、权限变更等）。这些技能是独立的、可组合的基础能力单元。

支持通过配置文件声明式编排这些技能。用户可按需选择要加入的技能并自定义执行顺序，像拼装积木一样灵活定义治理工作流，彻底脱离硬编码。

配置文件语法直观，用户无需编写代码即可定义新工作流。工作流 = 技能 A → 技能 B → 技能 C，通过调整顺序和组合即可应对不同业务场景。

## 业务价值

治理场景的交付从"代码开发"转变为"配置组装"，新工作流上线周期大幅缩短。用户可以像搭积木一样自由组合技能，无需等待开发资源。

原子技能统一收敛了基础能力，消除重复代码。相同的原子技能在不同工作流中复用，维护成本降低。

创始人可自行通过配置快速试错新流程，完全脱离对开发资源的依赖。业务需求变化时，可以立即调整配置文件验证效果。

## 成功标准

原子技能库覆盖所有已知的基础治理操作，每个技能独立测试、可复用。

任何新的治理工作流均可 100% 通过配置文件组装实现，无需编写代码。

配置语法直观，非技术人员能独立完成技能的选择与顺序编排。工作流执行结果可预期、可验证。
