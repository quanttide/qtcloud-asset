# 数字资产全景图

## 场景一：资产关系不透明

### 工作角色

查询方：需要了解资产全貌的人，如技术总监、新入职成员
维护方：负责资产更新的人，如文档负责人、模块负责人

### 问题

触发：新成员入职接手项目，或管理者需要汇报资产完整度。

现状：数字资产分散在不同系统中，资产间的关联关系存在于人的头脑中或零散的笔记里。一份架构设计文档引用了哪些代码模块、哪些文档已过期需要更新、流程规范的前置条件是什么，这些问题没有人能完整回答。

困难：老员工离职时带走的关键信息无法完整交接。新成员面对海量资料无从下手，需要花费数周时间才能理解资产结构。资产更新不同步，文档和代码经常对不上。

后果：人员流动导致知识断层，项目进度受影响。管理者做决策时信息不足，无法准确判断研发进度和风险点。每次汇报都需要人工整理资产清单，效率低下且容易出错。

### 假设

如果存在这样一个平台，能够自动识别和记录所有数字资产，提取资产间的引用、依赖、继承等关联关系，并以可视化的方式呈现全景图，让查询方能随时了解资产全貌和细节，维护方能及时同步资产变化。

## 场景二：资产变化难追踪

### 工作角色

变更方：修改或创建资产的人
关注方：依赖该资产的其他成员

### 问题

触发：资产发生变更，如文档更新、代码重构、流程调整。

现状：资产变化后，相关人员往往不知情。依赖该资产的其他成员继续使用旧版本，导致信息不一致。变更历史记录在各处，缺乏统一的追溯机制。

困难：无法快速了解某个资产的演进历史，不知道它从什么状态变成现在的样子。资产之间的影响范围难以评估，一处变更可能引发连锁反应但无人察觉。

后果：团队协作中出现版本混乱，重复劳动增加。关键变更信息丢失后，问题排查困难，同样的错误可能重复发生。

### 假设

如果存在这样一个平台，能够在资产变化时自动更新关联关系图谱，记录变更历史和影响范围，让关注方能及时获知依赖资产的变动，变更方能清晰看到自己的修改对全局的影响。
