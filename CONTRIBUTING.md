# 项目维护指南

本文档说明如何维护 `.quanttide/` 目录下的契约文件。

## 目录结构

`.quanttide/` 目录存放项目的契约定义，是项目的基础元数据。

```
.quanttide/
├── asset/
│   └── contract.yaml     # 资产组成契约
└── code/
    └── contract.yaml     # 代码规则契约
```

## 契约文件说明

### 资产契约 (asset/contract.yaml)

定义项目所有资产的组成和路径，包括文档、代码、部署配置、基础设施等。

**何时修改：**
- 新增或删除资产目录
- 资产路径变更
- 资产描述更新

**修改示例：**
```yaml
assets:
  new_asset:
    title: 新资产名称
    type: docs | code | config | experiment
    category: 分类标识
    path: 相对路径
    description: 简要描述
```

### 代码契约 (code/contract.yaml)

定义代码的编程规范、依赖关系、调用图和质量门禁。

**何时修改：**
- 引入新框架或语言
- 更换代码风格工具
- 模块间调用关系变更
- 质量门禁标准调整

**修改示例：**
```yaml
standards:
  module_name:
    language: 编程语言
    style: 风格规范
    linter: 检查工具

dependencies:
  module_name:
    framework: 框架和版本
    runtime: 运行环境
```

## 维护原则

### 契约优先

契约文件是项目的单一事实来源。实际代码和目录结构必须与契约保持一致。

### 变更同步

修改契约后，确保实际项目结构已同步更新。契约不是文档，是约束。

### 最小必要

只在契约中记录必要信息。不记录实现细节、不记录临时状态。

### 版本追踪

每次修改契约需要在提交信息中说明变更原因和影响范围。

## 审查清单

提交契约变更前检查：

- [ ] 路径是否真实存在
- [ ] 类型和分类是否准确
- [ ] 描述是否清晰无歧义
- [ ] 是否与现有契约冲突
- [ ] 提交信息是否说明变更内容
